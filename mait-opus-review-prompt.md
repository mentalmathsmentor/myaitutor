You are the architect. Your job is NOT to write the merged code. Your job is to:

1. Deeply review both codebases

## Step 2: Produce the Diff Report

Create a structured comparison covering:

### A. What Kimi adds (NEW — must be integrated)
List every new UI element, component, layout pattern, or content section that doesn't exist in the current codebase. For each, note whether it's purely visual or requires backend logic.

### B. What Kimi drops (EXISTING — must be preserved)
List every functional element from the current codebase that is missing or broken in Kimi's redesign. This is the critical section. Pay special attention to:
- Backend API call logic
- The clipboard-copy → open Gemini flow
- Syllabus dot-point data and how it's loaded into the prompt
- State variables that control worksheet generation parameters
- Route definitions and page structure
- Utility imports (privacy, keystroke, etc.)
- The "Generate Prompt & Launch Gemini" CTA behaviour
- Past papers page elements (THSC Online card, First Education card, exam timer)
- AI Resources page
- Free Demo / WebLLM integration
- Login/auth flow (if any)
- Feedback mailto: or form functionality
- Any easter eggs (there's a rickroll QR code in the exemplar PDF — preserve it)

### C. What conflicts (BOTH HAVE — needs resolution)
Where both codebases implement the same feature differently (e.g., topic selection UI, landing page layout, navigation structure), document both approaches and recommend which to use, or how to merge them.

### D. Styling reconciliation
Compare the design tokens, colour palettes, fonts, and Tailwind classes. Note any conflicts. The Kimi aesthetic should win for new components, but existing components should not look jarring alongside them.

## Step 3: Write the Sonnet Execution Prompt

Based on the diff report, write a complete, unambiguous prompt saved as `SONNET_MERGE_PROMPT.md` in the repo root. This prompt must:

### Structure
- Start with a brief context paragraph (what MAIT is, what we're doing)
- Include the full diff report as reference
- Then provide an ordered list of implementation tasks

### Task Format
Each task must specify:
- **What:** The exact change to make
- **Where:** The exact file path(s) to create or modify
- **How:** Enough detail that Sonnet doesn't need to make judgement calls. Include specific state variable names, import paths, component structures, and data flow descriptions where needed.
- **Preserve:** Any existing logic that must NOT be modified or removed during this task
- **Verify:** How to confirm the task was done correctly (e.g., "the worksheet generator should still copy the prompt to clipboard and open Gemini in a new tab")

### Task Ordering
Order tasks by dependency — foundational changes first (routing, shared state, utilities), then component-level changes, then page-level assembly, then styling cleanup, then testing.

### Critical Constraints for Sonnet
Include these rules at the top of the Sonnet prompt:
1. NEVER delete existing functional logic. If Kimi's design doesn't include something that the current codebase has, keep the current version and integrate it into Kimi's layout.
2. All backend API calls must be preserved exactly as they are. Do not modify endpoint URLs, request formats, or response handling.
3. The worksheet generation flow is: user selects options → prompt is assembled from selections + syllabus context → copied to clipboard → Gemini opens in new tab. This flow must work after the merge.
4. All embedded syllabus data (NESA dot-points, topic lists, outcome codes) must be preserved. If Kimi's design references different topic data, use the CURRENT codebase's data as the source of truth.
5. Maintain mobile responsiveness. The 4-step wizard must work on phone screens.
6. Do NOT create new backend endpoints. All changes are frontend-only.
7. After all changes, run `cd mait-mvp-glitch/frontend && npm run dev` and verify every route loads without errors.
